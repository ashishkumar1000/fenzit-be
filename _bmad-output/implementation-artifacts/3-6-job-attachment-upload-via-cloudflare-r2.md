# Story 3.6: Job Attachment Upload via Cloudflare R2

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a technician,
I want to upload photos and a customer signature for a job via a presigned URL,
so that there is proof of work captured before I mark the job complete.

---

## Upload Architecture (read before coding)

### Two-phase upload ŌĆö Free plan now, Worker-ready tomorrow

**Phase 1 ŌĆö Request presigned URL** (`POST /api/v1/jobs/:id/attachments`):
- Backend creates a row in the `attachment_uploads` **staging table** (`status: 'pending'`, `expires_at: now() + 15 min`) and returns a presigned PUT URL + the row UUID as `uploadId`.
- The client uploads the file **directly to R2** using that presigned PUT URL.

**Phase 2 ŌĆö Confirm** (`POST /api/v1/jobs/:id/attachments/:uploadId/confirm`):
- Client calls this after R2 responds 200 to the PUT.
- Backend validates the upload (not expired, belongs to this job/tenant, not already confirmed), then atomically: inserts a row into `attachments`, marks the staging row confirmed, and auto-advances `photos_uploaded` if this is the first photo.

**Why a staging table (not inline `pending` in `attachments`):**
- `attachments` only ever contains confirmed, real uploads ŌĆö the 5-photo gate, read URLs, and `getJobDetail` all work on clean data.
- Expired/abandoned staging rows are cheap to clean up without touching `attachments`.
- Parallel uploads (multiple photos at once) are safe ŌĆö each is a separate staging row, confirmed independently.

**Worker migration path (zero schema changes):**
When the account upgrades to Cloudflare Paid, skip Phase 2 from the client entirely. The Worker calls `POST /internal/webhooks/storage` which runs the same `confirm_attachment` RPC. The staging table, `attachments` table, and RPC are **identical** ŌĆö no migration needed, no backend changes. Just stop calling the confirm endpoint from mobile and deploy the Worker.

```
NOW (Free plan):
  Mobile ŌåÆ POST /api/v1/jobs/:id/attachments       ŌåÉ gets { presignedPutUrl, uploadId }
  Mobile ŌåÆ PUT {presignedPutUrl}  (direct to R2)
  Mobile ŌåÆ POST /api/v1/jobs/:id/attachments/:uploadId/confirm

FUTURE (Paid plan, Worker active):
  Mobile ŌåÆ POST /api/v1/jobs/:id/attachments       ŌåÉ same
  Mobile ŌåÆ PUT {presignedPutUrl}  (direct to R2)
  R2 ŌåÆ Queue ŌåÆ Worker ŌåÆ POST /internal/webhooks/storage  ŌåÉ same confirm_attachment RPC
  (Mobile no longer calls /confirm)
```

### One presigned URL per file ŌĆö no batching

Each photo is a separate request+confirm pair. The mobile client calls Phase 1 for each image independently (can be done in parallel), then confirms each independently. This is:
- Simpler retry logic ŌĆö one failed upload doesn't block others
- Naturally parallel ŌĆö 5 photos ŌåÆ 5 concurrent request+PUT+confirm cycles
- Consistent with how R2/S3 ecosystems work at scale

### R2 key pattern

`{tenant_id}/jobs/{job_id}/{folder}/{uuid}.{ext}`
- `folder`: `photos` for photos, `signature` for signature
- `uuid`: generated by backend at request time (stored in staging row)
- Extension derived from mimeType: `image/jpeg ŌåÆ jpg`, `image/png ŌåÆ png`, `image/heic ŌåÆ heic`
- Key encodes all routing metadata ŌĆö Worker can parse it without a DB lookup

## Acceptance Criteria

1. **Given** a valid `filename`, `mimeType` (`image/jpeg`, `image/png`, or `image/heic`), and `attachmentType: "photo"` on a job with fewer than 5 **confirmed** photos, **when** `POST /api/v1/jobs/:id/attachments` is called by the **assigned technician**, **then** HTTP 200 with `{ presignedPutUrl, uploadId, key, expiresAt }`. A row is created in `attachment_uploads` with `status: 'pending'`. The `key` follows `{tenant_id}/jobs/{job_id}/photos/{uuid}.{ext}`.

2. **Given** a valid `filename`, `mimeType`, and `attachmentType: "signature"`, **when** `POST /api/v1/jobs/:id/attachments` is called, **then** HTTP 200 with `{ presignedPutUrl, uploadId, key, expiresAt }` for the signature slot. Key: `{tenant_id}/jobs/{job_id}/signature/{uuid}.{ext}`.

3. **Given** a `mimeType` not in the allowed list, **when** `POST /api/v1/jobs/:id/attachments` is called, **then** HTTP 422 `VALIDATION_ERROR`.

4. **Given** a job already has 5 **confirmed** photos, **when** a 6th photo upload is requested, **then** HTTP 409 `DUPLICATE_RESOURCE` ("Maximum of 5 photos already uploaded").

5. **Given** a valid `uploadId` that belongs to this job/tenant and is not expired and not already confirmed, **when** `POST /api/v1/jobs/:id/attachments/:uploadId/confirm` is called by the assigned technician, **then** HTTP 200 with `{ id, type, createdAt }`; a row is inserted into `attachments`; the `attachment_uploads` row is marked `status: 'confirmed'`.

6. **Given** the confirmed attachment is a photo and it is the **first confirmed photo** for the job (photo count in `attachments` was 0 before this confirm), **then** the `photos_uploaded` workflow step is auto-advanced atomically: `jobs.current_step` updated + `activity_logs` entry inserted ŌĆö all in one `confirm_attachment` RPC call. Auto-advance only fires when `jobs.current_step = 'in_progress'` exactly.

7. **Given** a signature re-upload: a new presigned URL is requested and confirmed when a signature already exists for the job, **then** the existing signature row in `attachments` is replaced (upsert: UPDATE if exists, INSERT if not) ŌĆö no duplicate signature row is created.

8. **Given** a `uploadId` that is expired (`expires_at < now()`), **when** `POST /api/v1/jobs/:id/attachments/:uploadId/confirm` is called, **then** HTTP 410 `GONE` ("Upload session expired ŌĆö request a new presigned URL").

9. **Given** a `uploadId` that is already confirmed, **when** confirm is called again, **then** HTTP 200 with the original `{ id, type, createdAt }` (idempotent confirm ŌĆö same result, no duplicate insert).

10. **Given** `/internal/webhooks/storage` receives a POST with a valid `WORKER_WEBHOOK_SECRET` Bearer token and `{ key, size, tenantId, jobId, attachmentType }`, **then** HTTP 200; the same `confirm_attachment` RPC is called as in Phase 2 ŌĆö this endpoint is the Worker migration path and must remain functional. (On Free plan it is not called by any Worker; it can be tested manually.)

11. **Given** the webhook arrives with invalid/missing secret, **then** HTTP 401 `UNAUTHORIZED`.

12. **Given** an Owner JWT on `POST /api/v1/jobs/:id/attachments` or the confirm endpoint, **then** HTTP 403 `FORBIDDEN`.

13. **Given** a technician who is not the job assignee on either endpoint, **then** HTTP 403 `FORBIDDEN` (after 404 check).

14. **Given** a job id that does not exist or belongs to another tenant, **then** HTTP 404 `RESOURCE_NOT_FOUND`.

15. **Given** a malformed `:id` or `:uploadId` (not a UUID), **then** HTTP 400 (via `ParseUUIDPipe`).

16. **Given** a technician JWT with `tenantId: null`, **then** HTTP 400 `VALIDATION_ERROR`.

17. **Given** no `Authorization` header, **then** HTTP 401 `UNAUTHORIZED`.

18. **Given** `GET /api/v1/jobs/:id` on a job with confirmed attachments, **then** the `attachments` array is populated with real rows, each with a **freshly generated R2 presigned read URL (1-hour TTL, regenerated every call, never stored)**. Shape: `{ id, type, url, createdAt }`. (Resolves D1 deferred from Story 3.3.)

19. **Given** an `X-Idempotency-Key` UUID v4 header on `POST /api/v1/jobs/:id/attachments`, **when** the same key is replayed within 24 hours, **then** HTTP 200 with the original `{ presignedPutUrl, uploadId, key, expiresAt }` (reuses `IdempotencyInterceptor`).

## Dev Notes

### ŌÜĀ’ĖÅ Cloudflare Free Plan: Queues not available

R2 itself is free (10 GB storage, 1M Class A ops/month). Cloudflare Queues (needed for R2 ŌåÆ Worker event notification) **requires Workers Paid** ($5/mo). The Worker source is committed but the Queue binding is commented out. The `/internal/webhooks/storage` endpoint exists and is functional for manual testing now, and becomes the Worker's target when you upgrade.

### Why this design is Worker-ready

The `confirm_attachment` RPC is the single source of truth for "this upload happened". Both paths call it:
- **Client confirm** (now): `POST /api/v1/jobs/:id/attachments/:uploadId/confirm` ŌåÆ calls RPC with `p_actor_id = user.userId`
- **Worker** (future): `POST /internal/webhooks/storage` ŌåÆ parses `uploadId` from R2 key ŌåÆ calls same RPC with `p_actor_id = NULL`

When you upgrade: deploy the Worker, enable the Queue binding, set up the R2 event notification in Cloudflare dashboard, and remove the `/confirm` call from the mobile app. Zero backend migrations needed.

### Why a staging table (`attachment_uploads`)

The pattern comes from high-scale upload systems (used by Dropbox, Notion, Figma for their upload flows):

1. **`attachments` stays clean** ŌĆö only confirmed uploads; `getJobDetail`, photo count gate, and read URL generation all work on real data.
2. **5-photo gate is correct** ŌĆö counts confirmed photos, not abandoned pending URLs.
3. **Parallel uploads are safe** ŌĆö each photo gets its own staging row; confirming one doesn't affect others.
4. **Cleanup is cheap** ŌĆö a future cron can `DELETE FROM attachment_uploads WHERE status = 'pending' AND expires_at < now()` without touching `attachments`.
5. **Idempotent confirm** ŌĆö the RPC returns the existing row if already confirmed, so the mobile client can safely retry a confirm call.

### `sizeBytes` from the client

The presigned PUT URL response from R2 does not include the file size in a machine-readable header. The client knows the file size (it reads it before uploading). We trust it for Phase 1 ŌĆö it's not a security-critical value, and forging it has no meaningful impact. Note this in deferred-work for future verification if needed.

### No `photo_index` column

The original design included `photo_index` to track display order. This is dropped in favor of `created_at ASC` ordering:
- `photo_index` requires atomically assigning an integer at insert time ŌĆö tricky with concurrent uploads.
- If a photo is deleted, indices have gaps or need re-numbering.
- `created_at ASC` is stable, free, and sufficient for display ordering.
The mobile client should render photos sorted by `createdAt`.

### `updated_at` must be set explicitly in RPCs

`jobs.updated_at` has no `BEFORE UPDATE` trigger ŌĆö any `UPDATE jobs` in `confirm_attachment` must include `updated_at = now()`. [Source: `20260621000002_create_jobs.sql:25-27`]

### CRITICAL: `/internal/webhooks/storage` must NOT be under `api/v1`

Add `'internal/webhooks/storage'` to the `exclude` array in both `main.ts` and the e2e test bootstrap. NestJS matches the path string without a leading slash: `exclude: ['health', 'internal/webhooks/storage']`.

### Error code for expired upload

There is no `GONE` error code in `error-code.enum.ts`. Use `HttpException` directly:
```ts
throw new HttpException(
  { error_code: ErrorCode.RESOURCE_NOT_FOUND, message: 'Upload session expired' },
  HttpStatus.GONE,  // 410
);
```
Or add `UPLOAD_EXPIRED = 'UPLOAD_EXPIRED'` to `ErrorCode` enum and use it. Either is fine ŌĆö document the choice in deferred-work if not adding to enum.

### Idempotency on request endpoint only

`@UseInterceptors(IdempotencyInterceptor)` is on `POST /api/v1/jobs/:id/attachments` only. The confirm endpoint does NOT use it ŌĆö idempotency there is handled natively by the RPC returning the existing row if already confirmed. Adding the interceptor to confirm would cache the response but also block re-tries with the same idempotency key, which is wrong behavior for a confirm.
