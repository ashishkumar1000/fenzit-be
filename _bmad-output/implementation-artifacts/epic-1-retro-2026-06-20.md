## What Went Well

1. **Review-then-patch discipline** — every story went through code review. Findings were triaged into fix/defer, and all patch-grade items were applied before marking done. Story 1.6 had 5 patches applied post-review — none were dropped.

2. **`fromCallCount` mock pattern** — reliable pattern for mocking sequential Supabase `.from()` calls developed in Story 1.4 and reused across all subsequent unit and e2e tests. Prevented brittle over-specified mocks.

3. **Compensating deletes for partial failures** — Story 1.6 added a rollback on `user_skills` INSERT failure to avoid orphaned `users` rows. Good defensive design under non-atomic constraints.

4. **Deferred work is tracked explicitly** — security items were not swept under the rug. Every deferred finding has context in `deferred-work.md` and is traceable to the story that originated it.

5. **E2E tests maintained from the start** — Story 1.1 established the e2e test pattern; every subsequent story followed it. This provided real integration confidence throughout the epic.

---

## Challenges & Growth Areas

1. **`IsUUID('4')` test trap (Story 1.6)** — `IsUUID('4')` rejects test UUIDs with version nibble `0` (e.g. `00000000-0000-0000-0000-000000000001`). Cost debugging time. A quick `class-validator` doc lookup would have surfaced this immediately.

2. **Security consistently deferred** — RLS gaps, unauthenticated RPC access (`setup_tenant_for_owner`), `isValid = true` hardcoded OTP — all explicitly deferred per project direction. 22 items total in `deferred-work.md`. Real risk before production deployment.

3. **Non-atomic multi-step DB writes** — `inviteTechnician()` has 3 sequential round-trips with no transaction. Compensating deletes added but not a true rollback. Will compound as Epic 2+ adds more entity-creation flows.

4. **Re-invite flow missing** — an already-`invited` technician can't be re-invited; they receive a misleading 409 conflict. Known gap from Story 1.4, deferred.

5. **Library/framework behavior assumed rather than verified** — multiple moments where assumptions about `class-validator`, Supabase client chaining, or UUID specs were made without checking current docs. Slowed debugging.

---

## Key Insights

1. The `fromCallCount` pattern is robust but verbose. Epic 2 should consider whether a higher-level mock helper is worth abstracting.
2. Non-atomic multi-step writes are an architectural pattern risk — establishing a Postgres function / transaction discipline early in Epic 2 will prevent accumulated debt.
3. Deferred security items are all pre-production blockers — the hardcoded `isValid = true` OTP in particular must be resolved before any real user traffic.

---

## Team Agreement: MCP & Web Docs First

**Going forward:** Before assuming library or framework behavior, reach for docs first.

**When to use:**
- `class-validator` / `class-transformer` decorator behavior
- Supabase client chaining, RLS, error codes
- NestJS lifecycle questions (guards, pipes, interceptors)
- UUID standards, Postgres error codes, JWT spec
- Any time a test fails in a way that "shouldn't" happen

**Tools:** Context7 MCP for library docs → Supabase MCP for DB/schema questions → web search as fallback.
