## What Went Well

1. **Docs-first discipline landed** — Story 2.2 verified PostgREST `.or()` chaining, wildcard syntax, and unsanitized-interpolation risk against current docs before writing a line of query code. This directly informed the cursor injection fix and the sanitizer design. Epic 1 Action 1 applied in practice.

2. **Phone-split reconciliation before code was written** — Epic FR-13 specified single E.164 `phone`. The codebase had already pivoted to split `(country_code, phone_number)`. Story 2.1 dev notes caught this discrepancy in the create-story phase. Zero migration rework needed.

3. **Adversarial code review found real bugs every story** — no ceremony reviews. Both High-severity findings (23503→500, cursor injection) were caught post-implementation by the review process. The 3-layer format (blind/edge/auditor) is genuinely earning its cost.

4. **Reusable mock helpers established** — `mockInsert`, `mockQuery`, `mockSingle` give Epic 3 developers a clean mock cookbook. Each story built on the previous; by 2.3 the pattern was well-understood.

5. **Placeholder discipline was the right call** — `jobCount: 0`, `lastJobDate: null`, `jobHistory: []` all return valid typed empty values with NOTE comments pointing to Epic 3. The API contract is locked; Epic 3 fills data without changing types or signatures.

6. **Defense-in-depth tenant assertion added in review** — `data.tenant_id !== owner.tenantId` post-fetch check in `getCustomerDetail` ensures a future dropped `.eq('tenant_id')` filter could never leak a cross-tenant row. Added as a review patch; kept as a permanent guard.

7. **W4 whitespace-name bug proactively closed** — Story 2.1 applied the Epic 1 retro W4 lesson to the Customer DTO without being prompted. `@Transform(trim)` + `@IsNotEmpty()` on `name` confirmed by e2e test.

## Challenges & Growth Areas

1. **23503 FK violation → 500 not caught before review (High)** — A customer created with a syntactically valid but non-existent dial code (e.g., `+99`) passed the DTO regex, hit a FK constraint on `country_codes`, and returned 500 to the client. The developer handled `23505` (duplicate) but missed the FK table on the other end. Root cause: developer validated the value at the DTO layer and didn't model the DB-layer constraints exhaustively. Fix: catch `23503`, return 422 `VALIDATION_ERROR`.

2. **Cursor injection via forged `decodeCursor` not caught before review (High)** — `decodeCursor` validated type but not format. A crafted cursor like `{ id: "x),or(tenant_id.neq.0", createdAt: "..." }` decoded cleanly, then injected filter syntax into the unsanitized PostgREST `.or()` string — a potential tenant isolation bypass. Fix: UUID regex on `id`, ISO timestamp charset + `Date.parse` validation on `createdAt`. Lesson: PostgREST `.or()` unsanitized status must be treated as a first-class concern at every injection point, not just at the search term level.

3. **Guard ordering bug in `.single()` error handling (Story 2.3)** — Initial guard `(error?.code === 'PGRST116' || !data) → 404` masked genuine DB errors (e.g., `code: '08006'`) as 404 Not Found. A unit test explicitly covering the `{error: {code:'08006'}, data: null}` matrix cell caught this. Root cause: the developer read the Skills precedent but didn't extract the invariant correctly — the check order matters. Corrected pattern: check `error && error.code !== 'PGRST116'` → 500 FIRST; then `!data` → 404.

4. **Sanitizer over-stripped `.` and `:` (Story 2.2)** — Original strip regex `/[,().:*\\]/g` included `.` and `:`, breaking searches for names containing periods or colons (e.g., "St. John"). These characters are safe inside an ilike value; only the PostgREST filter-syntax characters need stripping. Fix: narrow to `/[,()*\\]/g`.

5. **Reading a reference pattern ≠ internalising an invariant** — Story 2.3 dev notes explicitly cited the `PGRST116` precedent in `skills.service.ts` and still got the guard ordering wrong. Citing a pattern prevents obvious omissions; it doesn't guarantee the subtle invariant is correctly applied. Explicit documentation of the invariant (not just the example) would help.

## Key Insights

1. **PostgREST `.or()` injection is a recurring, first-class security surface.** Any user-controlled data interpolated into a `.or()` string is an injection risk. This pattern will appear again in Epic 3 (job list filters, workflow queries). Explicit project-level documentation is needed so all Epic 3 developers carry this context by default.

2. **The PGRST116 guard ordering invariant is non-obvious and must be documented.** "Check genuine errors → 500 FIRST, then empty data → 404" is counterintuitive. It was cited by reference, applied incorrectly, caught by a unit test. An explicit code comment stating the invariant (not just pointing at an example) is more reliable than a cross-reference.

3. **`createAdmin()` bypasses RLS — RPCs must enforce tenant isolation internally.** C1 is deferred codebase-wide, but it becomes load-bearing in Epic 3. The `create_job_with_log` and `advance_workflow_step` RPCs run under the service role. If they don't validate `tenant_id` inside the function, and `createAdmin()` provides no RLS protection, tenant isolation rests entirely on the calling app layer. Epic 3 story dev notes must call this out explicitly.

4. **Placeholder strategy creates clean forward contracts.** Establishing `jobCount`, `lastJobDate`, `jobHistory` shapes in Epic 2 with typed interfaces and NOTE comments means Epic 3 has a contract to fill, not a design decision to make. This pattern should be continued whenever Epic N depends on Epic N+1 data.

5. **The 3-layer adversarial review earns its cost.** Every story had real bugs caught. Two were High severity (client-triggerable 500, potential tenant isolation bypass). The blind layer catches implementation holes; the edge layer catches matrix gaps; the auditor layer catches AC drift. None of these overlap cleanly. The format is worth keeping.

### New Technical Concerns for Epic 3

1. **Atomic RPCs are load-bearing** — `create_job_with_log`, `advance_workflow_step`, `increment_job_counter` must be atomic and must validate tenant isolation internally (service role bypasses RLS). Correctness of `increment_job_counter` (sequential `JB-YYYY-NNNN`, year rollover, race-free) is the hardest correctness problem in the project so far.

2. **Dual-role access is new** — Stories 3.2 and 3.3 are the first endpoints accessible by both Owner and Technician. Service layer silently applies `technicianId = user.userId` filter for Technicians (cannot be overridden by query param). Test coverage must verify both roles.

3. **404-not-403 semantic changes for Technician** — Story 3.3: cross-*tenant* access → 404 (same as Epic 2). Intra-tenant but wrong-technician access → 403. This is a different semantic than Epic 2's always-404 rule and must be explicitly documented in Story 3.3 dev notes.

4. **`IdempotencyGuard` (Story 3.5) blocks Story 3.6** — The guard must be built before Story 3.6's attachment endpoint can be fully tested. Story ordering 3.5 → 3.6 is mandatory.

5. **Cloudflare R2 (Story 3.6) requires external setup** — R2 bucket, Cloudflare Worker deployment, `WORKER_SECRET` env var, `wrangler.toml` queue configuration. Not blocking Stories 3.1–3.5; must be in place before Story 3.6 kickoff. Owner: Ashish.
